# regent — agent skill

`regent` is a grep/regex rule registry. Rules live as `*.rule.ts` files paired
with sibling `*.md` prose. The CLI scans code, emits findings as text or
SARIF, and supports tri-state review (pending → accepted/violation).

You (an agent) typically use regent to **scan a user's codebase for patterns
that don't fit Roslyn analyzers / ESLint / biome** — naming grep, structural
patterns, file-glob conventions, project-shape invariants.

## When to use

Use regent when the rule:
- is a single-line grep (RE2 syntax — no backrefs, no lookbehind)
- travels with prose (`.md` sibling) that explains the rationale
- needs to work across repos via user-global rules (`~/.agents/rules/`)
- produces output for both humans (text) and CI (SARIF)

Do **not** use regent for:
- AST-based analysis (use ESLint, biome, Roslyn)
- Cross-file invariants (regent is line-based)
- Compile-time type checking (tsc, Roslyn)
- Real-time editor feedback (regent is CLI-only)

## Invocation

```sh
regent check                          # scan git-changed files, report findings
regent check --all                    # scan entire scope
regent check --format sarif --out a.sarif  # SARIF for GitHub code scanning
regent check --no-review              # CI variant (no review candidates)
regent review                         # pending review findings, markdown
regent review --format json           # pending findings, JSON
regent review --include-accepted      # audit view (accepted + pending)
regent list                           # every loaded rule + origin
regent explain <rule-id>              # source path + rationale
regent init                           # scaffold tools/audit/ in current repo
regent accept <rule> <path>           # silence pending finding (--reason required)
regent reject <rule> <path:line>      # escalate pending to violation
regent llm                            # print this file
regent --llm                          # same, as flag
regent --version
regent --help
```

Exit codes: `0` no violations, `1` violations at or above `--exit-on`,
`2` usage error (missing `--reason`, malformed input).

## Writing a rule

Create `<rule-id>.rule.ts`:

```ts
import { defineRule } from '@dot-stbl/regent';

export default defineRule({
  id: 'csharp.no-region-directive',           // required, namespaced
  severity: 'error',                          // error | warning | suggestion
  pattern: '^\\s*#region\\b',                 // RE2 — no backrefs, no lookbehind
  excludeWhen: '\\boverride\\b',              // optional — skip lines matching this
  globs: ['**/*.cs'],                         // files to scan
  excludePaths: ['**/bin/**', '**/obj/**'],   // optional
  message: 'short human message',
  source: 'code-shape.md#no-region',          // optional, auto-derived
  rationale: 'longer explanation',            // optional
  review: {                                   // optional — tri-state mode
    enabled: true,
    exitBehavior: 'unreviewed-fails',         // or 'no-fail' (default)
    guidance: 'what the reviewer should check',
  },
});
```

RE2 syntax cheatsheet:
- `\\b` word boundary, `\\s` whitespace, `\\d` digit
- `^` start of line, `$` end of line
- `(a|b)` alternation, `(?:...)` non-capturing group
- `.*?` lazy, `[^x]` negated class
- NO backreferences `\\1`, NO lookahead `(?=...)`, NO lookbehind `(?<=...)`

Place rules in one of:
- `~/.agents/rules/<category>/<rule>.rule.ts` (user-global, auto-loaded)
- `<repo>/tools/audit/rules/<rule>.rule.ts` (repo-local, committed)
- `src/presets/<lang>.ts` (shipped with @dot-stbl/regent itself)

## Configuration

`<repo>/tools/audit/config.ts`:

```ts
import { defineConfig } from '@dot-stbl/regent';

export default defineConfig({
  extends: ['@dot-stbl/regent/presets/csharp'],
  rules: {
    disable: ['csharp.no-public-fields'],
    override: { 'csharp.no-private-methods': { severity: 'warning' } },
    accept: [
      {
        ruleId: 'csharp.no-todo-without-owner',
        path: 'src/Foo.cs',                  // glob; line optional
        // line: 42,
        reason: 'tracked in JIRA-123',        // required, max 500 chars
      },
    ],
  },
});
```

`<repo>/tools/audit/config.local.ts` — gitignored, per-developer layer.

## Tri-state review

Each finding has one of three states:

| State      | Trigger                                  | `check` shows? | `review` shows? | Fails CI? |
|------------|------------------------------------------|----------------|-----------------|-----------|
| `pending`  | review-rule matched, not accepted        | yes (section)  | yes             | only if `exitBehavior: 'unreviewed-fails'` |
| `accepted` | matched accept-list for `(rule, path[, line])` | no     | only with `--include-accepted` | never |
| `violation`| non-review rule OR unreviewed review-rule | yes (top)     | no              | when severity ≥ `--exit-on` |

## Layer merge (top wins)

1. Built-in presets (`@dot-stbl/regent/presets/*`)
2. User-global (`~/.agents/rules/**/*.rule.ts`)
3. Repository (`tools/audit/config.ts` + `tools/audit/rules/`)
4. Per-developer (`tools/audit/config.local.ts`, gitignored)

Same `ruleId` in multiple layers: top wins. `disable` removes; `override`
changes severity/message; `add` extends with new rules; `accept` silences
pending findings.

## How to handle findings

```sh
# Read the finding — note the ruleId, path, and the rule's guidance
regent review

# Either accept (silence with documented reason):
regent accept csharp.no-todo-without-owner src/Foo.cs \
    --reason "legacy file, tracked in JIRA-123"

# Or reject (escalate to violation, fails CI):
regent reject csharp.no-todo-without-owner src/Foo.cs:42

# Or fix the code, then re-run check
regent check
```

## Output formats

- **text** (default) — picocolors-coloured, multi-line context (3 lines
  before + 3 after), severity-coloured gutter. For terminals + agents.
- **sarif** — SARIF 2.1 with `region` + `contextRegion`. For GitHub code
  scanning via `github/codeql-action/upload-sarif@v3`.
- **both** — text + SARIF concatenated (separated by `--- SARIF ---`).
- **markdown** (`review` only) — bulleted findings with guidance.
- **json** (`review --format json`) — structured pending findings.

## Anti-patterns

- Don't write rules that need multi-line / cross-file analysis. Use ESLint
  or Roslyn for AST-level work; regent is line-based.
- Don't put secrets in `config.ts`. Use gitignored `config.local.ts` for
  personal accept entries / dev-only overrides.
- Don't disable rules globally — prefer `accept` with a reason. Disables
  are silent forever; accepts are auditable.
- Don't write rules without a sibling `.md`. The `.md` is the prose that
  the `.rule.ts` encodes — without it, future readers lose the rationale.
- Don't use RE2 backreferences / lookbehind. The matcher is `re2-wasm` and
  rejects those patterns at compile time.
- Don't run `regent check` against the regent repo itself unless you
  have a `tools/audit/config.ts` — without it, only the built-in presets
  apply, and `changed-only` mode means untracked rule files aren't scanned.
